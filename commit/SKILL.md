# Commit Skill

## 処理フロー

1. **変更の確認** — `git status` と `git diff` で未コミットの変更を確認
2. **変更がない場合** — 「コミットすべき変更がありません」と伝えて終了
3. **変更内容の分析** — staged/unstaged/untracked の変更を分析
4. **コミットメッセージの生成** — 変更内容に基づいてコミットメッセージ（タイトルのみ）を生成
5. **ユーザーに確認** — 生成したコミットメッセージをユーザーに提示し、承認を求める
6. **コミットの実行** — ユーザーが承認したら、変更をステージングしてコミットを実行

## コミットメッセージのルール

- **タイトルのみ**（本文は不要）
- **英語で記述**
- **Conventional Commits 形式**: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:` などの prefix を使用
- **70文字以内**
- **変更の「なぜ」を簡潔に表現**

## コミット実行時の注意

- `git add` で変更をステージングしてからコミット
- 機密ファイル（`.env`, `credentials.json` など）はステージングしない
- `Co-Authored-By` ヘッダーは付与しない
- コミット後に `git status` で結果を確認

## ユーザーへの確認フォーマット

```
以下のコミットメッセージでコミットします:

  <コミットメッセージ>

対象ファイル:
  - file1.ts (modified)
  - file2.ts (new)
```
